### why we write tsc --init

tsc --init কমান্ডটি আপনার প্রোজেক্টের রুট ফোল্ডারে একটি tsconfig.json ফাইল তৈরি করে।
এই ফাইলটি হলো TypeScript Compiler-এর জন্য নির্দেশিকা (Instruction Manual)। এতে লেখা থাকে:

কোন ফাইলগুলো কম্পাইল (ট্রান্সপাইল) করতে হবে?

কম্পাইল করার পর .js ফাইলগুলো কোথায় রাখতে হবে?

কোন ECMAScript ভার্সনে (ES5, ES2020) কম্পাইল করতে হবে?

কতটা কঠোরভাবে টাইপ চেক করতে হবে?

tsc --init মানে হলো— "TypeScript কম্পাইলারকে বলে দেওয়া, আমার কোড তুমি কোথায় বানাবে, কীভাবে বানাবে এবং কতটা কড়া হবে।"
আপনি আজকে এই ফাইলটি তৈরি করে ঠিক করে নিলেই সপ্তাহ ১-এর বাকি অথেন্টিকেশন কোড (যা আমি দেব) তা ঠিকমতো কাজ করবে এবং npx tsc চালালেই সব গুছিয়ে dist ফোল্ডারে তৈরি হবে।

### why need package.json file

package.json ফাইলটি হলো আপনার পুরো Node.js প্রজেক্টের "পরিচয়পত্র" (Identity Card) এবং "কন্ট্রোল সেন্টার" (Control Center)।

আপনি যদি এটি না রাখেন, তাহলে আপনার প্রজেক্টটি শুধু একটি ফোল্ডারেই থেকে যায়—কেউ জানে না এটি কী, এটি চালাতে কী কী লাগে, বা এটি কীভাবে শুরু করবে।

একেবারে সহজ বাংলায় বলছি, এই ফাইলটি ৫টি ম্যাজিক কাজ করে আপনার প্রজেক্টকে প্রফেশনাল করে

১. 📦 ডিপেন্ডেন্সি ট্র্যাকিং (কী কী প্যাকেজ লাগে?)
আপনি যখন npm install express prisma typescript লিখেছেন, তখন এই ফাইলটির ভেতরে dependencies এবং devDependencies নামক একটি তালিকা তৈরি হয়।

"এই তালিকায় লেখা থাকে—এই প্রজেক্ট চালাতে 'Express 4.18.2', 'Prisma 5.0.0' লাগবে।"

প্রোডাকশনে কেন দরকার? আপনি যদি আপনার কোড GitHub-এ পুশ করেন এবং অন্য কোনো ডেভেলপার (git clone) করে, সে শুধু npm install কমান্ড দিলেই আপনার সব প্যাকেজ অটোmatically ডাউনলোড হয়ে যাবে। তাকে আলাদা করে বলে দিতে হয় না কোন কোন প্যাকেজ লাগবে।

২. 📜 স্ক্রিপ্টস ম্যানেজমেন্ট (কমান্ড শর্টকাট)
আপনি হয়তো node server.js বা ts-node src/index.ts লিখে ক্লান্ত। package.json-এর "scripts" অংশে আপনি নিজের ইচ্ছেমতো ছোট কমান্ড বানাতে পারেন।

"scripts": {
"dev": "nodemon src/index.ts",
"build": "tsc",
"start": "node dist/index.js",
"db:generate": "prisma generate",
"db:migrate": "prisma migrate dev"
}

৩. 🏷️ প্রজেক্টের মেটাডেটা (নাম, ভার্সন, বিবরণ)
এখানে আপনি আপনার প্রজেক্টের নাম (SalesNexus), ভার্সন (1.0.0), ডেসক্রিপশন, অথর (আপনার নাম), এবং লাইসেন্স লিখে রাখতে পারেন।

"এটি বলার মতো বিষয় নয়, কিন্তু ইন্টারভিউয়ার যখন আপনার GitHub খুলে package.json দেখবে, এবং তাতে আপনার নাম, ইমেইল, এবং প্রজেক্টের উদ্দেশ্য সুন্দরভাবে লেখা থাকবে—তখন তার মনে হবে আপনি একজন অর্গানাইজড ডেভেলপার।"

৪. 🔒 package-lock.json-এর জন্ম দেওয়া (নির্ভরযোগ্যতা)
আপনি যখন npm install দেন, তখন package.json-এর পাশাপাশি package-lock.json ফাইলটি তৈরি হয়।
এটি কেন দরকার? package.json-এ সাধারণত ^4.18.2 লেখা থাকে (মানে 4.18.2 বা তার থেকে উপরের কোনো ভার্সন)। package-lock.json ঠিক করে দেয় "ঠিক 4.18.2 ইনস্টল করতে হবে"।
ফলে আপনি বা আপনার টিমের অন্য কেউ যখন কোড পুল করে, তখন সবার মেশিনে এক্স্যাক্টলি একই ভার্সনের প্যাকেজ ইনস্টল হয়। এর ফলে কারো মেশিনে "কাজ করছে, কারোতে করছে না" – এই সমস্যা প্রায় ৯০% কমে যায়।
